## 一、总体结论

结论：**方向正确，适合在现有 dashboard 上增量演进；但当前设计文档对“事件模型边界、派生指标口径、迁移/回填方案、系统通知接入协议”定义还不够严谨，若直接按现稿开工，第一阶段大概率会在数据契约上反复返工。**

我建议按 **“先统一事件协议与时间线查询，再做详情页，再做聚合分析”** 的顺序推进，避免前端先行后数据模型返工。

建议评级：
- **可立项性**：高
- **Phase 1 落地可行性（Flask + SQLite）**：高
- **当前文档可直接开发程度**：中
- **主要风险**：事件模型与现有 Chat/Watcher 协议不一致、阶段耗时模型过于线性、聚合表设计偏早且口径未固化

## 二、按审查重点逐项结论

## 2.1 数据模型设计（SQLite schema / 事件模型 / 索引策略）

### 2.1.1 现有基础可复用，方向正确
现有表：
- `tasks`
- `task_events`

已经足够承载：
- 任务主事实
- 任务状态流转基础时间线
- 当前看板 / 甘特图 / Agent 统计

因此“在现有 schema 之上增量扩展”是对的，不需要重做一套新库。

### 2.1.2 `communication_events` 是必须的，但字段定义还需收紧
方案里的 `communication_events` 思路正确，但建议补齐以下字段与约束：

#### 建议字段
```sql
communication_events (
  event_id TEXT PRIMARY KEY,
  task_id TEXT,
  thread_id TEXT,
  channel TEXT NOT NULL,             -- general / task / watcher / dispatch / direct_nudge
  event_type TEXT NOT NULL,          -- task_announce / question / answer / decision / task_done / notify / dispatch / nudge
  source_type TEXT NOT NULL,         -- human / system
  from_actor TEXT,
  to_actor TEXT,
  priority TEXT,
  severity TEXT,
  message_text TEXT NOT NULL,
  reply_to TEXT,
  source_file TEXT,
  source_line INTEGER,
  source_msg_id TEXT,
  related_artifact_path TEXT,
  happened_at TEXT,
  observed_at TEXT NOT NULL,
  payload_json TEXT NOT NULL DEFAULT '{}'
)
```

#### 原因
1. **仅靠 `payload_json` 不足以支撑筛选与排障**。`source_file / source_line / source_msg_id` 能保证可追溯。  
2. `event_id` 不能假定总能直接复用 `msg_id`。来自 watcher / dispatch / send-to-agent 的系统事件，未必天然有 `msg_id`。需要允许用 **稳定去重键**（如 source+path+line/hash）生成。  
3. `channel` 与 `event_type` 要拆开。`watcher` 是来源渠道，不是事件语义；`notify` / `dispatch` / `nudge` 才是语义。  
4. `task_id` 应允许为空，但要明确：**任务详情页默认只展示 task 级事件；general 频道消息只有在显式带 `task_id` 或被映射规则关联后才进入任务时间线**。

### 2.1.3 当前设计与现有 Chat 协议存在不一致，必须先对齐
现有 `scripts/send-chat.sh` / `scripts/lint-chat.sh` 允许的类型只有：
- `text`
- `task_announce`
- `task_done`
- `question`
- `answer`
- `decision`

而方案文档中已经预设：
- `notify`
- `direct_nudge`
- `severity`
- `source_type=system`

**这说明设计稿里的事件模型，已经超出了当前 Chat Hub A-Lite 协议。**

如果不先补协议，会出现两个问题：
1. 文档写得能 ingest watcher/system 事件，但实际生产脚本发不出来；
2. dev-2 会被迫在 ingest 层做“隐式猜测映射”，导致后续统计口径漂移。

**建议：先补一个“事件协议补充文档”，统一枚举与来源映射，再开 schema 与 ingest 开发。**

### 2.1.4 `task_stage_durations` 适合作为派生层，但当前模型过于线性
当前提议：
- `create_to_dispatch_seconds`
- `dispatch_to_ack_seconds`
- `ack_to_result_seconds`
- `result_to_review_seconds`
- `review_to_verify_seconds`
- `verify_to_close_seconds`

这个模型适合 MVP，但有一个重要局限：**它默认任务阶段只走一次线性路径。**

而真实流程里可能出现：
- review 驳回再提交
- verify fail 再修复
- blocked ↔ working 多次往返

如果只保留一行聚合值：
- 可以做首页摘要
- **不能做返工率、循环次数、阶段内多轮耗时分析**

#### 建议
- Phase 1 保留 `task_stage_durations`，用于详情页和基础图表；
- 但同时在设计上明确：**返工/重试分析的事实源仍然是 `task_events` / `communication_events`，不能只靠 durations 表。**
- 更稳妥的中期方案是补一个：
  - `task_stage_spans(task_id, stage_name, seq_no, started_at, ended_at, duration_seconds, source_json)`

这样后面做 review 循环数、verify fail 次数时不会被第一版表结构卡死。

### 2.1.5 `task_metrics_daily` 设计过早，而且 EAV 口径偏弱
当前方案：
```sql
task_metrics_daily (
  stat_date TEXT,
  project TEXT,
  metric_key TEXT,
  metric_value REAL,
  payload_json TEXT,
  PRIMARY KEY (stat_date, project, metric_key)
)
```

这是一种 **EAV 风格（metric_key / metric_value）** 设计，优点是灵活，缺点是：
- SQL 可读性差
- 前端很容易出现 metric key 拼写分裂
- 同一页面要读多个指标时需要多次 pivot
- 很难对某些指标建立明确约束和单测

#### 建议
- **不要在 Phase 1 就落库 daily metrics。**
- Phase 2 开始前，先确认首页/分析页到底要展示哪些固定指标；
- 若指标集合已经比较稳定，优先用 **宽表** 或 **只读 SQL view**，例如：
  - `overview_metrics_daily`
  - `agent_metrics_daily`
- 如果仍想保留弹性，建议仅把不稳定的维度细节放进 `payload_json`，而不是所有值都做 EAV。

### 2.1.6 `agent_metrics_daily` 的 `role` 字段语义不够严谨
当前设计：
```sql
agent_metrics_daily (
  stat_date TEXT,
  agent_id TEXT,
  role TEXT,
  ...
)
```

问题：在当前团队体系里，一个 agent 在不同任务里可能承担不同职责：
- assigned_agent
- reviewer
- integration_owner
- owner_pm（虽然通常不是 agent）

所以这里的 `role` 如果只有一个字段，很容易语义失真。

#### 建议
二选一：
1. **去掉 `role`，只保留 `agent_id` 指标**；
2. 或者改为更明确的：
   - `participation_role`（assigned / reviewer / integration / qa）
   - 并允许一名 agent 同一天多行。

### 2.1.7 索引策略应以“查询形态”驱动，避免机械全字段建索引
拆解文档提议对 `communication_events` 至少给这些字段建索引：
- `task_id`
- `thread_id`
- `from_actor`
- `to_actor`
- `event_type`
- `happened_at`

方向上没错，但如果全部做单列索引，收益未必最高，写入成本会升高。

#### 更建议的索引
```sql
CREATE INDEX ix_comm_task_time ON communication_events(task_id, happened_at);
CREATE INDEX ix_comm_thread_time ON communication_events(thread_id, happened_at);
CREATE INDEX ix_comm_to_time ON communication_events(to_actor, happened_at);
CREATE INDEX ix_comm_from_time ON communication_events(from_actor, happened_at);
CREATE INDEX ix_comm_type_time ON communication_events(event_type, happened_at);
```

如果后续一定要按 `severity` 做风险面板，再考虑：
```sql
CREATE INDEX ix_comm_severity_time ON communication_events(severity, happened_at);
```

不建议一开始就给每个低选择性字段建独立索引。

### 2.1.8 必须补 migration / backfill 设计
现有 `dashboard/db.py` 只有：
- `SCHEMA_VERSION = 1`
- `CREATE TABLE IF NOT EXISTS ...`

**没有真正的版本迁移逻辑。**

而方案里已经默认：
- 旧库可升级
- 可兼容 backfill

这部分目前并不存在，必须显式补任务：
1. schema version 升级策略
2. 老库增表/加索引流程
3. 全量回填命令
4. 幂等重建策略

否则“旧数据库升级后仍能工作”这个验收标准没有实现基础。

## 2.2 技术选型是否恰当（Flask + SQLite）

### 2.2.1 Phase 1 / Phase 2 继续使用 Flask + SQLite 是合理的
基于现状：
- 当前只有本地/内网 dashboard
- 数据源是本地文件系统 (`tasks/`, `chat/`, watcher 事件)
- 请求主要是读，写主要来自 ingest/同步脚本
- 任务量与消息量总体仍在“小团队协作”量级

因此：
- **Flask 足够支撑 API 层**
- **SQLite 足够支撑事件存储与聚合查询**

现有实现已经启用：
- WAL
- busy timeout

这些都适合当前场景。

### 2.2.2 但要明确边界：它适合“团队运营分析”，不适合“高并发协作平台”
Flask + SQLite 的前提是：
- ingest 为批处理/串行或低并发
- dashboard 以只读查询为主
- 不做多人同时高频写入
- 不做秒级实时推流

如果未来演进到：
- 多人同时打开看板高频筛选
- watcher / chat / dispatch 同时高频落事件
- 需要复杂多维长周期统计

那时可能要考虑：
- Postgres
- 后台聚合作业
- 分页 / 缓存 / 物化视图

但**现在不需要超前升级**。

### 2.2.3 建议加两条实现约束，确保 SQLite 不被误用
1. **所有 ingest 统一走脚本或后台作业，不在 HTTP 请求里做重写/回填**；  
2. **所有复杂聚合先做“预计算或视图化”，不要让首页请求直接全表扫描 JSON payload。**

## 2.3 现有 dashboard 代码复用策略是否合理

### 2.3.1 “在现有 dashboard 上扩展”是正确路线
当前已有：
- `dashboard/app.py`：Flask API 骨架
- `dashboard/query.py`：查询组装
- `dashboard/ingest.py`：任务事实 ingest
- `dashboard/templates/index.html`
- `dashboard/static/js/dashboard.js`

现有 dashboard 已具备：
- 看板
- 甘特图
- Agent 统计

所以方案强调“增量演进，不另起系统”是合理的。

### 2.3.2 但前端不能继续把所有逻辑堆进单个 `dashboard.js`
当前 `dashboard/static/js/dashboard.js` 还是一个单文件脚本，结构适合 3 个简单 tab，不适合再长出：
- 总览
- 详情抽屉
- 沟通时间线
- 风险面板
- 角色分析

#### 建议
Phase 1 就拆分前端模块，至少拆成：
- `api.js`
- `board-view.js`
- `task-detail-view.js`
- `timeline-view.js`
- `analytics-view.js`（可后续）
- `helpers.js`

否则后续 UI 需求一上来，前端维护成本会快速上升。

### 2.3.3 后端应保持“查询层集中”，不要把 SQL 分散到 app.py
当前 `dashboard/app.py` 结构是好的：
- 路由薄
- query.py 负责组装 payload

这个边界应该保留。新接口应继续放到：
- `dashboard/query.py`
- 或分拆成 `query_tasks.py / query_analytics.py`

不建议在 Flask 路由里直接拼复杂 SQL。

### 2.3.4 最佳复用方式：保留现有 API，不新增破坏性替换
建议：
- 保持现有 `/api/board`、`/api/gantt`、`/api/agents` 不变；
- 新增：
  - `/api/tasks/<task_id>`
  - `/api/tasks/<task_id>/timeline`
  - `/api/tasks/<task_id>/communications`
  - `/api/analytics/overview`
  - `/api/analytics/agents`
  - `/api/analytics/bottlenecks`

这样能保证现有看板不回归，Phase 1/2 可以逐步叠加。

## 2.4 任务拆解的依赖关系和执行顺序

## 2.4.1 大方向基本正确
文档已经抓住了第一阶段的核心主线：
1. 事件模型
2. schema / ingest
3. 详情 API
4. 详情 UI
5. 联调验证

整体顺序是对的。

## 2.4.2 需要调整的依赖点

### 问题 A：PH1-05 的依赖不够完整
当前：
- `PH1-05` 只依赖 `PH1-02`

但实际上它还依赖：
- `PH1-01`（因为 watcher / dispatch / nudge 的 event_type、source_type、severity 都要先定义）

如果没有统一协议，PH1-05 很容易各脚本各写各的事件格式。

#### 建议
`PH1-05` 依赖改为：
- `PH1-01`
- `PH1-02`

### 问题 B：PH1-07 不应被 PH1-06 阻塞太久
当前：
- `PH1-07` 依赖 `PH1-01`、`PH1-06`

但 API payload 的大部分契约（基本信息、责任链、时间线、communication summary）其实可以先定。  
如果等阶段耗时完全落表再定 API，会平白阻塞 dev-1 / dev-2。

#### 建议
拆成：
- **PH1-07A**：先定任务详情 payload 主契约（依赖 PH1-01）
- **PH1-07B**：补充 durations 字段（依赖 PH1-06）

至少在执行顺序上，允许 `PH1-07` 提前启动。

### 问题 C：PH1-09 与 PH1-08 可以并行，不必完全串行
当前：
- `PH1-09` 依赖 `PH1-07`
- `PH1-10` 依赖 `PH1-08`、`PH1-09`

这是合理的，但 `PH1-09` 的 UI 设计不必等后端 API 完工。  
只要 payload 契约定了，UI 交互草图就可以并行设计。

### 问题 D：缺少“schema migration / backfill”显式任务
当前拆解里没有单独任务负责：
- schema version 升级
- 老库回填验证
- rebuild 路径

这会让 dev-2 在 PH1-02 / PH1-04 中隐式承担额外设计，风险较高。

#### 建议新增任务
- **PH1-02A**：设计并实现 schema migration / rebuild 策略

### 问题 E：缺少开发侧自动化测试任务
当前第一期只有：
- `PH1-14 qa-1 联调验证`

但缺少：
- ingest 单测
- query/payload 单测
- 前端 timeline transform 单测

#### 建议新增任务
- **PH1-08A**（dev-2）：补详情 API / timeline 查询单测
- **PH1-12A**（dev-1）：补时间线渲染与空态单测

否则 QA 会变成第一道而不是最后一道防线。

## 2.5 明显的架构风险或遗漏

### 风险 1：系统通知来源没有统一事实源
方案里希望 ingest：
- `dispatch-task.sh`
- `send-to-agent.sh`
- `task-watcher.sh`

但当前这些脚本并没有一个现成统一的事件日志面：
- `dispatch-task.sh` 负责派发与校验
- `send-to-agent.sh` 负责 tmux 发消息
- `task-watcher.sh` 负责通知和状态推进

它们可以发消息，但**未必稳定留下结构化事件文件**。

#### 风险
如果直接让每个脚本各自“顺手写一段 JSONL”，后面会出现：
- 字段名不一致
- 失败重试格式不一致
- 去重困难

#### 建议
不要让 3 个脚本各自定义格式。新增统一落盘入口，例如：
- `scripts/append-comm-event.py`
- 或 `scripts/chat-board-sync.py emit-event ...`

所有系统侧事件统一走这个入口。

### 风险 2：任务详情时间线需要一条统一排序规则
`task_events` 与 `communication_events` 合并后，要明确排序逻辑：
1. 优先 `happened_at` / `event_at`
2. 无事件时间时回退 `observed_at`
3. 同一时间戳的 tie-breaker 用 `event_id`

否则时间线很容易“看起来乱序”。

### 风险 3：`general` 频道消息与 task timeline 的关系未定义
现有 Chat Hub 有：
- `chat/general/`
- `chat/tasks/`

如果 `general` 里的决策或问题涉及某任务，但没显式带 `task_id`，那在任务详情页是无法自动归并的。

#### 建议
在设计里明确：
- **任务详情时间线只消费 task thread + 显式带 task_id 的通用消息**；
- 不做模糊文本匹配归因。

### 风险 4：阶段耗时与返工统计会冲突
当前 durations 模型适合“首轮线性耗时”；而管理层未来关心的是：
- 返工率
- review 循环
- verify fail 重试

如果不提前声明“durations 只是摘要视图”，后面业务方会误以为它能支撑全部分析。

### 风险 5：未明确分页/限流策略
任务详情页的沟通时间线未来可能很长。  
当前文档默认一次全拉，但后续可能出现：
- 单任务上百条消息
- 多任务切换频繁

#### 建议
API 预留：
- `limit`
- `before`
- `after`
- `event_types`

即使 Phase 1 不做分页 UI，后端也最好预留分页能力。

### 风险 6：现有 schema 缺少 foreign key / nullable 策略说明
`communication_events.task_id` 若要允许为空，就要明确：
- 是否对有值的 task_id 建立 FK 到 tasks
- 当 task 被删除/归档时如何处理历史事件

建议：
- task_id 可空
- 有值时逻辑上指向 tasks.task_id
- SQLite FK 可不强绑，避免 general 事件/孤儿事件写入失败
- 但查询层要把“找不到任务”的情况当成合法数据处理

## 2.6 具体改进建议

## 2.6.1 先补三份小契约文档，再开工
建议在开发前先补：

1. **事件协议补充文档**
   - event_type 枚举
   - source_type 枚举
   - severity / priority 枚举
   - watcher / dispatch / direct nudge 的映射规则

2. **任务详情 API 契约文档**
   - `/api/tasks/<task_id>`
   - `/api/tasks/<task_id>/timeline`
   - `/api/tasks/<task_id>/communications`

3. **迁移/回填策略文档**
   - schema version 变更
   - 老库升级
   - 全量 backfill
   - rebuild-all 语义

这 3 个补完后，第一期开发会顺很多。

## 2.6.2 推荐的第一期最小闭环
比原方案再收敛一点，我建议 Phase 1 MVP 顺序如下：

### Step 1：统一事件协议
- 定义 communication event 枚举
- 定义 system event 落盘入口

### Step 2：扩 schema + migration
- `communication_events`
- `task_stage_durations`
- schema version 升级 / rebuild

### Step 3：接 chat/tasks 与 system events ingest
- 先 ingest `chat/tasks/*.jsonl`
- 再接 watcher / dispatch / direct nudge
- `chat/general` 放第二优先级

### Step 4：做任务详情 API
- 先做基本信息 + 状态时间线 + communication timeline
- durations 可后补

### Step 5：做详情抽屉 UI
- 先完成单任务 drill-down
- Overview / Analytics / Alerts 先不扩太多

这条路径能最快把“任务全过程可视化”跑通。

## 2.6.3 推荐的查询层形态
建议新增一个统一时间线视图或统一查询组装器，而不是让前端自己 merge 两类事件：

- 后端把 `task_events + communication_events` 合并成统一 timeline payload
- 前端只负责渲染

这样能把：
- 排序
- 去重
- 类型映射
- 空态兜底

都收敛在后端，避免前后端各维护一份口径。

## 2.6.4 对 daily metrics 的建议：后移一拍
不建议第一阶段就落：
- `task_metrics_daily`
- `agent_metrics_daily`

更稳的做法是：
1. 先把详情页和时间线做对；
2. 再基于真实查询需求，决定哪些指标值得物化；
3. 优先把首页真正要展示的少数指标定死，再设计 daily table。

## 2.6.5 对复用策略的建议：保守扩后端，主动拆前端
- **后端**：保留 `app.py + query.py + ingest.py` 主体，不重构大框架；
- **前端**：尽早拆 JS 模块，不要继续单文件膨胀。

这是这轮改造里性价比最高的一处“提前投资”。

## 四、最终判断

### 可以保留的部分
- 在现有 dashboard 上增量扩展
- Flask + SQLite 作为 Phase 1/2 底座
- 任务详情页作为第一优先级
- communication events 作为新增核心模型

### 必须先补强的部分
- 事件协议与现有 Chat/Watcher 脚本的一致性
- schema migration / rebuild 策略
- durations 只是摘要，不是完整返工事实源
- daily metrics 不宜过早固化
- 任务拆解需补开发侧测试与迁移任务

### 一句话结论
> 这套方案的主方向是对的，但真正决定成败的不是“多几个图表”，而是先把 **任务事件、沟通事件、系统通知事件** 的协议和时间线语义统一好；只要这一层收紧，Flask + SQLite 完全足以支撑第一、二阶段的分析需求。
