# 功能级共享上下文协作规范

## 问题

当前每个 agent 被隔离在自己的任务目录里，只有 task.json 和 instruction.md（PM 单向写入），agent 之间没有沟通渠道。导致：
- 修了 A 不知道 B 也依赖同一套机制，修完引入新问题
- Review 只看当前 diff，审不出跨任务的回归
- 同一个问题反复修、反复退，越滚越大

## 核心思路

**先定义功能，再拆子任务；功能下建共享工作区，所有角色读写同一份上下文。**

## 功能目录结构

```
features/<feature-id>/
├── BRIEF.md          # 功能定义
├── CONTEXT.md        # 共享上下文（架构关系、影响范围、已知坑）
├── decisions.log     # 关键决策记录
├── tasks/            # 子任务目录（沿用现有 task 结构）
│   ├── fix-toolcall-chain/
│   └── frontend-status/
└── notes/            # 各角色备注
    ├── dev.md        # 开发发现的问题、踩过的坑
    ├── arch.md       # 架构补充、约束说明
    └── qa.md         # 测试发现、回归问题
```

## 文件职责

### BRIEF.md（功能定义，PM 写，所有人读）
- 功能名称和目标
- 验收标准
- 边界和约束（什么不做）
- 优先级和关联功能

### CONTEXT.md（架构师写，所有人读+写）
- 涉及哪些模块/文件
- 模块间的依赖关系
- 跟其他功能的耦合点
- 已知的敏感区域（改这里会影响什么）

### decisions.log（所有人追加）
格式：`[时间] [角色] 决策内容 — 原因`
避免同一个问题被反复讨论或推翻。

### notes/（所有人读+写）
- **dev.md**：开发过程中发现的坑、临时方案、需要后续清理的东西
- **arch.md**：架构层面的补充说明、约束变更
- **qa.md**：测试发现的回归问题、边界 case

## 工作流程变更

### 1. 需求进来时（PM）
- 创建 `features/<feature-id>/` 目录
- 写 BRIEF.md
- **不要立即拆子任务**，先让架构师写 CONTEXT.md

### 2. 架构师分析（arch-1）
- 读 BRIEF.md
- 写 CONTEXT.md（影响范围、模块依赖、敏感区域）
- 给出建议的任务拆分（可以是一个大任务，也可以是多个子任务）
- **如果改动范围小且明确，建议合并为一个任务，不走完整审查流程**

### 3. 任务拆分（PM）
- 基于 BRIEF + CONTEXT 决定拆几个任务
- **bug fix / 小改动**：一个任务，走轻流程（dev → 冒烟测试 → 合入）
- **新功能 / 架构变更**：可拆多任务，走完整流程
- 每个子任务的 instruction.md 开头必须注明：`功能目录：features/<feature-id>/`，要求 agent 先读 CONTEXT.md 和 notes/

### 4. 执行（dev）
- **开始前**：读 BRIEF.md + CONTEXT.md + notes/
- **过程中**：发现问题及时更新 notes/dev.md
- **完成后**：更新 notes/dev.md 记录改了什么、有没有遗留

### 5. 审查（review-1 / arch-1）
- **开始前**：读 CONTEXT.md，了解全局影响
- **审查时**：不仅看 diff，还要检查是否影响 CONTEXT.md 中标注的关联模块
- 审完更新 notes/arch.md

### 6. QA（qa-1）
- **开始前**：读 BRIEF.md（验收标准）+ notes/（已知问题）
- 不仅测当前任务，还要回归 CONTEXT.md 中标注的关联功能
- 完成后更新 notes/qa.md

## 任务粒度判断指南

| 场景 | 建议 | 流程 |
|------|------|------|
| 明确的 bug fix | 一个任务，一个 dev | dev → 冒烟 → 合入 |
| 小功能（<3 模块） | 一个任务 | dev → review → QA → 合入 |
| 中功能（3-5 模块） | 1-3 个子任务 | 架构师 CONTEXT → dev → review → QA → 合入 |
| 大功能 / 架构变更 | 按模块拆，但共享上下文 | 架构师 CONTEXT → 并行 dev → 集成 → review → QA → 部署 |

# task-watcher 优化方向

基于今天实际运行中暴露的问题，按优先级排列。

## 1. 进程级可靠性（最高优先级）

今天卡死了两次，没有任何自愈，整个自动流转链条中断。

**方案：**
- 主循环每轮更新一个 heartbeat 时间戳文件（如 `.task-watcher/heartbeat`）
- 加一个外层 watchdog（cron 或 launchd），检测时间戳超过 2 分钟未更新则杀掉重启
- 或直接用 launchd 的 `KeepAlive` 托管，进程退出自动拉起
- 心跳已配置在 HEARTBEAT.md 中每 30 分钟检查，但 watcher 本身的自愈才是根本

## 2. 超时检测粒度优化

当前 dispatched 超时 60 秒就重发指令，但 agent 可能在思考中（Codex 响应较慢），导致误报"超时未确认"推到飞书。

**方案：**
- 区分"无响应"（无 ack，超过 3 分钟）和"工作中"（有 Working 标记）
- "工作中"但超过 30 分钟：通知 PM 介入，不重发
- "无响应"超过 3 分钟：重发指令
- 重发前先检查上次重发时间，避免短时间内重复重发

## 3. 通知去重和幂等

当前用 `_route` 标记文件判断是否已通知，但重启后标记文件还在，可能跳过本该重新触发的事件。

**方案：**
- 标记文件里存 signature（文件 mtime），只有签名变了才重新触发
- `review_signature` 逻辑已有雏形，扩展到 result/verify 的通知判断
- 删除标记文件时用 TTL（如 1 小时后自动过期），避免标记无限累积

## 4. 通知失败重试

`push_feishu` 当前是 fire-and-forget（`&` + `disown`），失败无记录。

**方案：**
- 关键通知（推给林总工飞书的）失败时写错误日志
- 网络问题简单重试一次
- 非关键通知（agent 间 tmux 通知）失败可忽略

## 5. 日志持久化

当前日志只输出到 tmux pane，重启后丢失，无法事后排查。

**方案：**
- 重要事件（状态变更、通知发送、错误）追加写到日志文件
- 保留最近 7 天，自动轮转
- 日志文件位置：`~/.openclaw/workspace/logs/task-watcher.log`

## 6. 轮询效率（远期）

每次轮询扫所有 tasks/ 目录，任务多了会慢。

**方案：**
- 短期不急，当前任务量下 5 秒轮询够用
- 后续任务量上来可改成 inotify/file event 驱动（如 fswatch）

## 建议实施顺序

1. **进程可靠性 + 日志持久化** — 解决今天的核心痛点
2. **超时检测优化** — 减少误报
3. **通知去重 + 重试** — 提升通知可靠性
4. **轮询效率** — 按需
