# 任务集成队列与分批合入方案

> 日期：2026-05-27
> 范围：并行开发任务的 patch / branch 交付、integration 分支合入、冲突回写、任务最终收口。

## 背景

当前任务执行链路已经能为开发任务准备独立 worktree / branch，并在 `result.json` 写入时生成 `patch_path`。这保证了 agent 之间可以并行开发，但也留下一个缺口：任务完成后只留下 patch 作为交付证据，代码并不会自动合入目标分支，往往还需要单独创建“合入任务”。

这个缺口会带来三类问题：

- `ready_for_merge` 不等于代码已经进入可验证主线，任务状态和代码状态容易脱节。
- 多个 patch 堆积后，越晚合入越容易出现冲突，且冲突责任不够结构化。
- PM / integrator 要靠人工记忆逐个补合入任务，控制面不能形成闭环。

## 目标

- 保留 patch 作为审计、复现和失败回放证据。
- 新增标准集成入口，把 `patch_path` / `workspace_branch` 合入 `integration_target_branch`。
- 并行开发、串行合入：同一目标分支一次只处理一个任务。
- 冲突不静默吞掉，必须写入 `integration.json` 并把任务转为可处理状态。
- 默认不自动推送远端；只有显式 `--push` 才执行 `git push`。

## 非目标

- 不让普通开发 agent 直接合主干。
- 不在第一版自动直推 `main`。
- 不让 dashboard 直接改 `task.json`。
- 不在本轮实现长期后台 daemon；先提供可由 integrator / cron / watcher 后续调用的标准脚本。

## 状态与工件

新增任务工件：

```json
{
  "task_id": "任务ID",
  "status": "pass",
  "source_type": "branch",
  "source_ref": "task/xxx",
  "target_branch": "integration",
  "integration_commit": "abc123",
  "integrated_by": "integrator",
  "integrated_at": "ISO-8601",
  "tests": [],
  "summary": "已合入 integration"
}
```

失败时：

```json
{
  "task_id": "任务ID",
  "status": "fail",
  "reason": "merge_conflict",
  "conflict_files": ["src/foo.py"],
  "stderr": "截断后的 git 输出"
}
```

`task.json` 更新规则：

| 集成结果 | `task.json.status` | `merge_gate_state` | 说明 |
|---|---|---|---|
| pass | `done` | `closed` | 代码已进入目标分支，本任务收口 |
| fail | `blocked` | `blocked` | 需要 PM / 原 agent 处理冲突或失败 |
| dry-run | 不落盘 | 不落盘 | 只输出计划和检查结果 |

## 合入策略

默认策略为 `auto`：

1. 如果 `workspace_branch` 在仓库中存在，优先 merge branch。
2. 否则使用 `patch_path` 做 `git apply --3way --index`。
3. 两者都缺失时 fail-close，写明 `integration_source_missing`。

目标分支来自：

1. CLI `--target-branch`
2. `task.json.integration_target_branch`
3. `task.json.target_branch`

第一版要求目标分支必须已经存在于本地或远端；是否创建 `integration` 分支应由初始化/运维动作显式完成，避免脚本误建错误基线。

## 冲突处理

- 纯文本冲突：integrator 可在后续修复任务中处理，或重新运行脚本。
- 语义冲突：任务转 `blocked`，`integration.json.conflict_files` 给原 agent 作为返修上下文。
- 契约冲突：PM / 架构师重新排序或拆分任务。

原则：开发并行，合入串行；小冲突可由 integrator 消化，大冲突回任务闭环。

## 第一版落地

1. 新增 `scripts/integrate-task.py`
   - 单任务合入。
   - 支持 `--dry-run`、`--test-cmd`、`--push`。
   - 成功写 `integration.json(status=pass)` 并收口任务。
   - 失败写 `integration.json(status=fail)` 并阻塞任务。

2. 新增 `scripts/integrate-ready-tasks.py`
   - 串行扫描 `ready_for_merge` 任务。
   - 只处理质量 gate 已满足、且存在 branch/patch 的任务。
   - 默认遇到失败即停止，防止在未知坏基线后继续批量合入。

3. 更新 README / scripts README
   - 明确“patch 不是最终合入”。
   - 推荐 integrator 使用集成队列。

4. 补充回归测试
   - branch 合入成功并更新目标分支。
   - merge 冲突写入 `integration.json` 并阻塞任务。
   - dry-run 不落盘。

## 操作入口

```bash
# 批量预检：不改任务、不改目标分支
scripts/integrate-ready-tasks.py --dry-run

# 批量串行合入：默认遇到第一处失败即停止
scripts/integrate-ready-tasks.py --test-cmd 'scripts/teamctl.sh smoke'

# 单任务预检 / 合入
scripts/integrate-task.py --task-dir tasks/<任务ID> --dry-run
scripts/integrate-task.py --task-dir tasks/<任务ID> --test-cmd 'scripts/teamctl.sh smoke'
```

默认不推远端；需要发布目标分支时显式追加 `--push`。`--dry-run` 是单任务独立预检，不模拟整批任务累计合入后的最终状态；真正分批冲突由串行合入过程逐个暴露并停队列。
