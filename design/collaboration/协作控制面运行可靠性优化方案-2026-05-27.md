# 协作控制面运行可靠性优化方案

> 日期：2026-05-27
> 范围：`scripts/` 控制面、Chat Hub、task-watcher/watchdog、teamctl 启动链路、飞书同步脚本。

## 目标

- 降低新机器、非默认 checkout 路径、自定义 agent id 下的运行失败概率。
- 防止外部同步脚本在缺省配置下误操作真实飞书 Base。
- 让 README 推荐命令与实际 `teamctl` 启动行为一致。
- 用回归测试覆盖本次发现的真实失效路径，而不是只做脚本语法检查。

## 改动计划

1. 飞书同步安全边界
   - 移除任务详情同步中的硬编码 Base/Table 默认值。
   - 要求显式配置任务详情表 ID。
   - 增加 dry-run 模式，跳过所有飞书写入/删除动作。

2. 启动命令解析
   - `teamctl start-agents` 支持 `CODEX_CMD='codex -p dev-team'` 这类带参数命令。
   - 启动前只检查命令首个可执行文件，实际启动时保留完整命令行。

3. watchdog 宽限期
   - 修复 grace marker 写入时读取未设置 `GRACE_FILE` 的问题。
   - 用真实 watchdog 单次运行场景覆盖 heartbeat pid mismatch。

4. watcher 自动派发
   - 自动派发消息使用当前任务目录，不再拼接本机绝对路径。
   - 指定开发 agent 时按 config 中的开发角色判断，不再只认 `dev-*` 前缀。

5. Chat Hub 发送者归因
   - 从 `agents/<agent-id>` 路径层级或 config workdir 反推出发送者。
   - 避免 agent 在子目录执行 `send-chat.sh` 时把 `src`、`docs` 等目录名写进 `from`。

## 验收标准

- `bash -n scripts/*.sh scripts/lib/*.sh` 通过。
- 关键 Python 文件 `py_compile` 通过。
- 新增/更新的定向 pytest 覆盖：
  - watchdog grace marker 真实写入。
  - Chat Hub 子目录发送者归因。
  - task-watcher 自定义开发 agent 自动派发。
  - `teamctl` 带参数 runtime command 启动。
  - 飞书同步脚本无硬编码真实 ID，缺任务详情表 ID 时 fail-close。

## 非目标

- 本轮不重构整个 `task-watcher.sh` 状态机。
- 本轮不迁移 tracked `config.json` 到 `config.example.json` 模式；这是更大的发布/兼容性调整。
- 本轮不清理历史归档文档中的旧绝对路径示例。
