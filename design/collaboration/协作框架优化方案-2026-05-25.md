# 协作框架优化方案（2026-05-25）

> 创建日期：2026-05-25  
> 状态：PM 正式方案（待后续实现拆分）  
> 适用范围：`/Users/linsuchang/Desktop/work/my-agent-teams` 的任务派发、agent 协作、task-watcher、任务池、通信渠道、dashboard 与控制面脚本  
> 方案来源：基于 2026-05-24 ~ 2026-05-25 一轮真实任务推进、重启恢复、working 超时、ready_for_merge 收口、blocked 仲裁、QA 旁路与任务池排查后的 PM 复盘结论。

---

## 0. 一句话结论

当前协作框架的核心问题，不是 agent 数量不足，也不是缺少任务池 / watcher / dashboard，而是：

```text
输入缺口 + 控制面状态脏化 + 可消费库存不可见
```

因此优化重点不是继续加功能，而是优先把以下三件事工程化：

1. **控制面状态一致性自动修复**
2. **外部阻塞与内部阻塞分层表达**
3. **让 PM / agent 一眼看清“哪些任务真的能推进”**

---

## 1. 本轮复盘确认的主要问题

## 1.1 任务状态存在，但真实可推进性不透明

虽然已有：
- `pending`
- `pooled`
- `dispatched`
- `working`
- `ready_for_merge`
- `blocked`
- `done`

但 PM 真正判断时仍要额外人工确认：
- `result.json` 是否存在
- `review.json / verify.json` 是否合法
- `assigned_agent / claim_scope / claimed_by / reserved_by` 是否一致
- watcher 是否已经把 gate 推进到正确阶段
- agent 会话是否真的恢复可用

这意味着：

> 当前状态机只能表达“名义状态”，还不能稳定表达“当前能不能推进”。

---

## 1.2 watcher 有能力，但控制面易出现脏状态

本轮真实出现过：
- `assigned_agent != claimed_by`
- `assigned_agent != reserved_by`
- `claim_scope` 不覆盖当前执行者
- `result.json` JSON 合法但 `status` 取值非法
- `working` 超时后任务多次 ack，但没有任何有效产出
- `ready_for_merge` 实际已 review pass，但卡在 QA / gate 未推进

说明问题不是“没有自动化”，而是：

> 自动化已经能流转，但缺少稳定的状态清洗与一致性收口层。

---

## 1.3 pooled 库存与真实吞吐脱节

当前任务池中经常存在大量 `pooled` 任务，但多数不可派发，原因包括：
- 依赖上游仍是 `pooled`
- 上游已经 `blocked`
- 外部样本源 / 挂载 / owner 输入缺失

结果就是：

```text
任务池有库存 ≠ 当前有可消费库存
```

这会误导 PM 对吞吐的判断。

---

## 1.4 blocked 语义过粗，无法快速判断谁该行动

当前 `blocked` 既可能代表：
- 代码未修完
- 工件非法
- QA 未通过
- 任务定义不完整
- 样本源缺失
- 外部挂载缺失
- owner 决策未给

这导致 PM 还要进一步人工解释：
- 是应该转派给 dev / qa / review
- 还是应该找 owner 补输入
- 还是应该冻结整个下游链

---

## 1.5 PM 当前承担了太多“状态修理工”职责

这轮 PM 实际花费大量时间在：
- 清理 claim / reserve 残留字段
- 修 `task.json` 一致性
- 修 `result.json` 非法状态值
- 手工 close `ready_for_merge`
- 转派 working 超时任务
- 替 QA/Review 解释“这是输入缺失，不是执行失败”

说明目前的框架把大量“控制面异常处理”压在 PM 身上。

---

## 2. 优化目标

## 2.1 目标

本轮优化的目标不是推翻重做，而是让现有机制收敛为：

```text
L1: task.json + artifact JSON = 唯一事实源
L2: watcher / reducer / router = 确定性编排层
L3: chat / send-to-agent = 通信与升级层
L4: dashboard / pool view / PM inbox = 只读观测层
```

要实现的具体效果：
- PM 一眼看出哪些任务真实可推进
- watcher 尽可能自动修复脏状态
- blocked 能区分内部可解 vs 外部输入缺失
- pooled 任务池能区分“库存”与“可消费库存”
- QA / Review 不再接到还不可审、不可验的任务
- agent 重启后有统一复位协议
- PM 高频治理动作脚本化

## 2.2 非目标

本轮不追求：
- 引入新数据库或复杂分布式队列
- 让 Chat Hub 成为任务状态事实源
- 通过提高 `working` 并发来掩盖依赖阻塞问题
- 一次性重构全部 watcher 逻辑

## 3. 正式优化方案（按优先级）

# P0：必须先做（稳定性与可推进性）

## P0-1. 引入任务健康态 `task_health`

### 目标
补足 `status` 不能表达“当前能否推进”的缺口。

### schema 落地
在 `task.json` 新增：

```json
{
  "task_health": "healthy",
  "health_reasons": []
}
```

建议枚举：
- `healthy`
- `artifact_invalid`
- `state_inconsistent`
- `input_missing`
- `session_unhealthy`
- `dependency_blocked`
- `waiting_gate`

### watcher 落地
在 `task-watcher.sh` 或其抽出的 reducer 中增加统一 health 归约：
- 先算 `status`
- 再算 `task_health`
- 把细节写入 `health_reasons`

例如：
- `ready_for_merge + review 已过 + verify 缺失` → `waiting_gate`
- `result.json.status=completed` → `artifact_invalid`
- `assigned_agent != claimed_by` → `state_inconsistent`
- blind holdout 缺样本源 → `input_missing`

### dashboard 落地
- 看板中增加 health badge
- PM Inbox 支持按 `task_health` 聚合
- 任务详情页显示 `health_reasons`

### 流程落地
PM 与 watcher 的默认动作按 health 走，而不是只看 status。

---

## P0-2. watcher 增加“状态清洗器”

### 目标
把常见可自动修复的脏状态前移到 watcher，而不是反复由 PM 手工修。

### 脚本落地
新增独立脚本：
- `scripts/task-state-sanitizer.py`

负责：
1. 清理 `assigned_agent != claimed_by/reserved_by`
2. 修复 `claim_scope` 不含当前执行者
3. 检测旧轮次 `ack/review/result/verify` 是否应归档
4. 检查 artifact 状态值是否合法
5. 对 `ready_for_merge` 缺 gate 条件的任务补写明确 `merge_gate_state`

### watcher 落地
`task-watcher.sh` 每轮：
1. 先跑 sanitizer
2. 再做 reducer / router / close / notify

### schema 落地
在 `task.json` 增加：

```json
{
  "sanitized_at": "ISO-8601",
  "sanitizer_actions": ["clear_stale_claimed_by", "align_claim_scope"],
  "state_invariant_violation_count": 0
}
```

### dashboard 落地
在任务详情中显示：
- 最近一次 sanitizer 动作
- 最近一次 state invariant violation

### 流程落地
PM 只处理 sanitizer 无法自动修复的问题。

---

## P0-3. 将 blocked 正式拆成 internal / external

### 目标
让系统明确知道：这条 blocked 是团队内部能解，还是缺 owner/外部输入。

### schema 落地
在 `task.json` 新增：

```json
{
  "blocker_type": "external",
  "blocker_code": "missing_unseen_source_pdfs",
  "blocker_owner": "owner",
  "unblock_requirements": [
    "提供新的未入库原始 PDF 来源",
    "给出绝对路径或挂载点"
  ]
}
```

建议：
- `blocker_type`: `internal | external`
- `blocker_owner`: `dev | qa | reviewer | pm | owner | infra`

### watcher / reducer 落地
- `verify fail` 不再一律理解为执行失败
- 若 `result/verify` 明确给出外部输入缺失，应自动归类为 `external`
- 对 `external blocked` 下游 pooled 任务直接标记为 `dependency_blocked(external)`

### dashboard 落地
- blocked 页签分列：`internal blocked` / `external blocked`
- PM Inbox 顶部优先显示 `external blocked`

### 流程落地
- `internal blocked`：转派 / 返工 / 重试
- `external blocked`：等待 owner 输入，不再反复催执行者

---

## P0-4. QA / Review 路由前置硬校验

### 目标
避免 reviewer / QA 接到还不可审、不可验的任务。

### 脚本落地
增强：
- `scripts/task-queue-router.py`

增加硬校验：
- `result.json` 存在
- JSON 合法
- `result.status` 在允许集合内
- 必要字段齐全
- 当前任务确实进入对应 gate
- artifact 轮次不落后于当前执行轮次

### watcher 落地
若未通过校验：
- 不推入 review/qa 队列
- 改写 `task_health=artifact_invalid` 或 `state_inconsistent`
- 给 PM 一个一次性待处理提示

### dashboard 落地
队列页展示：
- `queue_ready`
- `queue_blocked_by_invalid_artifact`

### 流程落地
review/QA 的注意力只花在真正 ready 的任务上。

## P1-2. 建立 agent 重启复位协议

### 目标
解决重启后 PM 手工探测、频繁转派的问题。

### 脚本落地
新增：
- `scripts/agent-recover-check.sh`
- 或 watcher 启动时支持 `recover-agent --agent <id>`

协议流程：
1. agent 重启上线
2. 自动扫描自己名下 `working/dispatched/blocked`
3. 回写恢复报告，例如：

```json
{
  "agent": "qa-1",
  "recovered_at": "ISO-8601",
  "tasks": [
    {
      "task_id": "整理BlindHoldout样本集并形成清单工件",
      "session_restored": true,
      "workspace_restored": true,
      "mount_restored": false,
      "recommended_action": "keep_blocked"
    }
  ]
}
```

4. watcher / PM 决定：
- `resume`
- `keep_blocked`
- `reroute`

### dashboard 落地
显示：
- 最近一次 agent 恢复时间
- 恢复后未确认任务数

### 流程落地
agent 重启不再只是“上线了”，而是自动进入复位检查流程。

## P1-3. communication 渠道分层与升级路径标准化

### 目标
减少 PM 作为“人肉路由器”的负担。

### 流程落地
明确五层职责：
1. `task artifacts`：唯一事实源
2. `chat/tasks/*.jsonl`：任务讨论与补充说明
3. `watcher system events`：状态推进与异常告警
4. `send-to-agent.sh`：催办、转派、硬提醒
5. `owner/alert-card`：外部输入、关键决策、生产风险

### 文档落地
新增或修订：
- `design/chat-hub/protocol.md`
- `design/collaboration/feature-shared-context.md`
- `design/agent-templates/*.md`

写清楚：
- 哪类信息必须落 artifact
- 哪类信息允许只发 chat
- 哪类情况必须升级到 PM / owner

### watcher 落地
- 对同类问题优先发 system event
- 达到阈值再升级为 direct nudge
- 达到 `external blocked` 条件则升级为 PM / owner

## 6. 风险与迁移注意事项

### 6.1 不要一次性改完所有状态字段
建议采用“新增字段 + 兼容旧字段”的方式迁移，不要直接破坏旧 watcher 逻辑。

### 6.2 sanitizer 先做只修小问题
P0 先让 sanitizer 自动修复：
- claim / assign / scope 不一致
- 非法 artifact 状态值
- 旧轮次残留工件

更复杂的自动决策暂缓。

### 6.3 blocked external 需要明确定义触发条件
否则容易把本应内部处理的问题错误上推给 owner。

### 6.4 dashboard 必须展示“为什么不可派发”
否则只是换一种方式继续误导 PM。

## 7. 结论

这套框架已经具备正确的骨架：
- 有任务事实源
- 有 watcher 自动编排
- 有 pooled 任务池
- 有 review / qa 队列
- 有 dashboard 和 chat 辅助观察

当前最需要做的不是继续堆功能，而是：

```text
先把控制面状态收敛干净
再把 blocked / pooled / queue 的真实语义讲清楚
最后把 PM 的高频治理动作工具化
```

优先级判断：
- **P0** 解决“状态脏、不可判断、不可自动修”的问题
- **P1** 解决“看不清、恢复难、升级路径乱”的问题
- **P2** 再去做“批量治理、统计分析、自动建议”

如果按这个顺序推进，当前框架不需要推翻，就能明显降低 PM 运维负担，并提升真实协作吞吐。

## 8. 补充：Review 流程缺口与闭环优化

结合 2026-05-25 对 3 个 P0 任务（`引入任务健康态与健康原因归约`、`实现任务状态清洗器并接入Watcher`、`区分内部外部阻塞并打通阻塞可见性`）的排查，可以确认当前 review 自动流转存在以下机制缺口。

### 8.1 现象判断
- 这 3 个任务都已经进入 `ready_for_merge`
- `merge_gate_state=quality_pending`
- `review_gate_state=pending`
- `qa_gate_state=passed`
- `quality_gate_mode=parallel`

说明：
- 不是任务没达到 review 条件
- 不是 review queue router 选不出候选
- 真正卡点在 watcher 到 reviewer 的“自动消费闭环”没有做完整

### 8.2 根因一：watcher 只负责入队 / 续推，不负责 review 执行闭环
当前 watcher 在任务进入 `quality_pending` 后，会将任务纳入 review 队列，并尝试给 reviewer 发送续推通知。

但 review 流与主任务流不同：
- 主任务有 `dispatched -> ack -> working -> result`
- review 流基本只有“入队 / nudge / 等待最终 review.json”

这意味着 watcher 只完成了“让 reviewer 看得到任务”，并没有形成“review 已接单、处理中、完成”的强状态闭环。

结果就是：
- PM 看到任务已经 `ready_for_merge`
- queue 里也能看到它
- 但 review 实际没有进入稳定的执行态
- 最终表现为“应该自动进入 review，但实际上停住了”

### 8.3 根因二：queue_state 是单 reviewer 串行占位，容易把后续任务堵住
当前 watcher 会为 reviewer 维护类似 `review-queue-<agent>.json` 的当前占位状态。

这个模型的问题是：
- 一个 reviewer 同时只能挂一个 current task
- 只要这个占位没被清掉，watcher 就不会继续自动 push 下一个 review task
- 后续 ready_for_merge 的任务虽然进入了 queue，但只能“排队可见”，不能“继续被主动推进”

这次 3 个 P0 任务的表现正符合这个模型：
- 第一个任务已被 watcher 挂到 review 占位
- 后两个任务能被 queue-router 列出来
- 但没有继续形成自动 review 推进

### 8.4 根因三：review 流缺少中间状态，PM / watcher 都无法判断卡在哪
当前 review 侧缺少明确的生命周期状态，例如：
- `review_assigned`
- `review_ack`
- `reviewing`
- `review_done`

没有这些中间状态，就会导致：
- PM 看不出 reviewer 是否真的接单
- watcher 看不出是“未送达”还是“送达未处理”
- dashboard 只能看到 `review_gate_state=pending`，但无法解释 pending 的原因

所以现在的 review pending 语义过粗，既包含：
- 等待派发
- 已派发未确认
- 已确认处理中
- reviewer 掉线/重启后失活

这会直接削弱 PM 的治理效率。

### 8.5 根因四：缺少僵尸占位回收机制
当前 queue_state 的回收主要依赖终态工件（例如 review.json）或任务 gate 变化。

但实际协作里经常会出现：
- reviewer 重启
- 消息送达但未消费
- review 会话丢失上下文
- reviewer 被切走处理别的任务

在这些情况下，如果没有：
- 超时回收
- idle 回收
- agent 重启回收
- 二次续推 / 转派策略

就会形成“僵尸占位”：
- queue_state 还在
- watcher 认为 reviewer 手上有活
- 但 reviewer 事实上没有继续处理
- 后续 review 队列整体被串行卡死

### 8.6 对优化方案的补充要求
基于以上根因，原方案应补充以下 P0 / P1 项：

#### P0 补充
1. 为 review 流增加显式中间状态：`review_assigned / review_ack / reviewing / review_done`
2. watcher 改为“入队 + 等待 ack + 超时重推/回收”的闭环模式
3. 在 queue_state 中记录 `assigned_at / last_seen_at / acked_at / source_task_id`
4. dashboard / PM inbox 显示“review pending 的具体原因”

#### P1 补充
5. 引入 reviewer 重启后的 queue reclaim 机制
6. 引入 review 超时升级：提醒、转派、PM 仲裁
7. 区分“queue visible”和“actively being reviewed”两种状态
8. 为 review/qa 队列补充可观测指标：排队时长、占位时长、僵尸占位数

### 8.7 最终定性
本次问题不是单点 bug，而是 review 阶段的控制面语义不完整：

```text
watcher 现在更像“把任务放进 review 队列并提醒 reviewer”
而不是“确保 review 生命周期真实推进并自动闭环”
```

因此后续实施时，必须把 review/qa 队列从“可见性机制”升级为“可消费、可回收、可追责的执行机制”。
