# 设计任务派发前空闲确认与非打断通知机制

> 目标：先确认角色空闲，再派发执行类消息；忙时不直接打断，改为排队、摘要或延后触达。

## 1. 现状问题复盘

当前“消息发出去就等于可以立刻执行”的假设不成立，主要断点有四个：

| 位置 | 现状 | 结果 |
| --- | --- | --- |
| `scripts/task-watcher.sh::is_idle_agent()` | 只看 `dispatched/working` 数量和 tmux 里是否有 `Working` 文案 | `pre_reserved_*`、`ready_for_merge`、`ack/result` 收尾态都可能被漏判 |
| `scripts/task-watcher.sh::send_session_message()` | 默认走 `send-to-agent.sh`，失败后回退 legacy tmux | 任何 routine 消息都可能绕过统一门禁 |
| `scripts/send-to-agent.sh` | 发送后若没看到响应文案会重试 | “其实已送达但 agent 当时没回” 会被误判成失败并重复发送 |
| `scripts/task-watcher.sh` / `dispatch-task.sh` | 续推、预留、首发都是按事件直接发 | 同一 agent 忙时仍可能被打断 |

另外，`scripts/lib/task_watcher_notifications.sh::push_feishu()` 只对飞书外部推送做了 `uuid` 去重，不能解决 agent 侧 tmux 消息重复；控制面必须在更早的位置做 event-key 去重。

## 2. 统一 busy / idle 口径

建议不要只分“忙/闲”，而是分成三档：

| 档位 | 典型状态 / 信号 | 是否允许执行类消息 | 是否允许提醒类消息 | 处理方式 |
| --- | --- | --- | --- | --- |
| `idle` | 无 `dispatched/working`、无 active `pre_reserved_*`、无收尾中任务 | 允许 | 允许 | 直接派发 |
| `soft_busy` | `ready_for_merge`、`review_pending`、`qa_pending`、`blocked`、`review_rejected`、`qa_failed`、`result_pending` | 不建议直接打断 | 允许摘要 / 延后 | 进队列，等空闲后合并触达 |
| `hard_busy` | `dispatched` 未 ack、`working`、active `pre_reserved_*`、当前任务仍占用 reserved slot | 禁止 | 仅留队列 | 不直发，禁止 routine tmux 打断 |

判定输入必须来自任务状态，而不是只看 tmux：

- `status`
- `ack.json` / `result.json` 是否存在
- `pre_reserved_by` / `pre_reserved_until`
- `reserved_by` / `claimed_by`
- `review_gate_state` / `qa_gate_state`
- 任务是否仍占用同一 reserved slot

`working_limit` / `reserved_limit` 只应当作为容量约束，不要拿来直接推断 idle。

## 3. 消息类型分级

| 消息类型 | 示例 | 默认路由 |
| --- | --- | --- |
| 执行类 | 新任务派发、自动续推、认领后开始执行 | `idle` 才直接发；其他情况入队 |
| 提醒类 | ack 提醒、result 提醒、review/QA 提醒、timeout 催办 | `idle` 直接发；`soft_busy` 摘要化；`hard_busy` 只记录队列 |
| 预热类 | 下一条任务预留、pre-reserve 提示 | 忙时默认只排队，不打断当前主线 |
| 广播类 | PM 公告、任务池总览 | 不面向单个 agent 的消息可直接发，但不要伪装成执行指令 |
| 例外强打断 | 生产事故、owner override、紧急 review fail 仲裁 | 允许绕过 busy gate，但必须显式声明并留审计 |

需要新增控制面 `event_key`，作为统一去重键：

```text
event_key = sha256(task_id + event_type + target_agent + task_state_version + round)
```

规则：同一 `event_key` 只允许“排队一次、发送一次、确认一次”。

## 4. 唯一判定入口应该放在哪一层

**推荐唯一硬门禁放在 `scripts/send-to-agent.sh`，并由一个共享 helper 提供 busy 判定。**

理由：

1. 它是所有 agent 定向消息的最后公共出口，最适合做“发出去之前先看忙不忙”的最终拦截。
2. `task-watcher.sh`、`dispatch-task.sh`、PM 手工入口都可以统一收敛到这个 wrapper，而不是各自实现一套判断。
3. `task-pool-router.py` 只负责“选谁接”，不负责“现在能不能发”；busy 逻辑放 router 会把选择和触达耦合死。

推荐结构：

```text
classify_agent_availability(agent_id)
  -> idle / soft_busy / hard_busy
classify_message_kind(kind)
  -> execute / remind / preheat / broadcast / critical
route_delivery(agent_id, message)
  -> direct / defer / digest / override
```

`task-watcher.sh::send_session_message()` 应该统一调用这个 wrapper；legacy tmux fallback 只能作为显式诊断/故障恢复路径，不能作为 routine 默认路径。

## 5. 非打断派发时序

```text
router 选出候选任务
  ↓
helper 判定 agent_availability
  ├─ idle        → 直接 send-to-agent
  ├─ soft_busy   → 写入 notification queue + 汇总 digest
  ├─ hard_busy   → 只写入 queue，不触发 tmux
  └─ critical    → 显式 override + 审计后直发
  ↓
watcher 在“agent 变空闲”或“任务状态进入收尾完成”时 drain queue
  ↓
同一 agent 的重复通知按 event_key 合并成一条摘要
```

关键点：

- `send-to-agent.sh` 的“没看到响应文案就重试”不能再作为语义级重发依据；
- 发送成功但暂时没回显，应记为 `delivery_unknown`，交给队列和下一次空闲检查处理；
- 不要用一次 shell 重试，把同一条执行指令发成两条。

## 6. 例外与 override 语义

允许打断 busy agent 的情况只应当是显式、可审计的少数场景：

- P0 / 生产事故
- owner override
- 紧急 review fail / 仲裁回退

需要满足：

- 显式 flag：`--force` / `--interrupt=critical`
- 必填原因：`override_reason`
- 必填操作者：`override_actor`
- 必填目标：`target_agent` / `task_id`
- 必填审计落点：`override_audit_event`

例外只允许“绕过 busy gate 一次”，不允许把队列和去重逻辑也一起绕过。

PM 手工发送任务消息也应默认走 busy-aware wrapper；直接 `tmux send-keys` 只应留给真正的救火场景。

## 7. 与 pre-reserve / working_limit / claim pool 的兼容规则

这一层必须和“预热任务独占预留机制”保持一致：

- `pre_reserved_*` 写入即计入 reserved slot，因此应视为 `hard_busy`；
- `dependency_ready -> dispatched` 是同一 slot 的继承转换，busy gate 不能把“本来就占着的 slot”再算一遍；
- `working_limit` / `reserved_limit` 仍然是容量门禁，不是消息门禁；
- `claim pool` 只决定“谁可以认领”，busy gate 决定“现在要不要触达”；
- 当前任务的续推、预留、派发都只能占用一个事件/一个 slot，不能在队列里重复计数。

简化理解：

- `pre_reserved_by=<agent>`：说明这个 agent 已经被预热占坑，不能再被 routine 消息打断；
- `dispatched/working`：说明这个 agent 正在执行，必须等空闲或显式 override；
- `ready_for_merge`：说明主线已结束，但收尾未完全闭环，默认只做摘要/延后，不直接打断。

## 8. 最小实现拆分建议

| 任务名 | 优先级 | 负责人建议 | write_scope | 估时 | 依赖 |
| --- | --- | --- | --- | --- | --- |
| 统一 Agent 可用性判定 helper | P0 | 平台脚本维护者 | `scripts/lib/agent_availability.sh`、`scripts/task-watcher.sh` | 2-4h | 无 |
| send-to-agent busy-aware wrapper | P0 | 消息派发维护者 | `scripts/send-to-agent.sh`、`scripts/lib/task_watcher_notifications.sh` | 3-5h | helper |
| watcher 自动续推 / 预留改为排队 + digest | P0 | watcher 维护者 | `scripts/task-watcher.sh` | 4-6h | helper + wrapper |
| dispatch-task 首发 / 重试节流 | P1 | dispatch 维护者 | `scripts/dispatch-task.sh` | 2-4h | wrapper |
| 通知队列与 digest 汇总 | P1 | 平台脚本维护者 | `scripts/lib/task_watcher_notifications.sh`、`scripts/task-watcher.sh` | 4-8h | helper + watcher |
| override 审计与 PM 手工入口 | P0 | PM 工具维护者 | `scripts/send-to-agent.sh`、`scripts/task-watcher.sh` | 2-3h | helper |
| pool view / router 暴露 busy_level | P2 | 看板维护者 | `scripts/task-pool-view.py`、`scripts/task-pool-router.py` | 1-2h | helper |

## 9. 结论

这套机制是可行的，而且应当尽快落地：

1. 先做统一 busy-aware helper；
2. 再把所有 agent 定向消息收口到 `send-to-agent.sh`；
3. 忙时只入队 / 摘要，不再直接打断；
4. 只保留少量显式 override 作为例外；
5. 预热、claim、reserved slot 与当前预热机制保持同一套 slot 语义。

这样可以把“派发前先确认空闲”从口号变成统一控制面规则，并且能直接拆成后续执行任务。
