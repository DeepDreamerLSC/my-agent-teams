# 多智能体协作流程优化方案 v2

> 基于 2026-04-24 首日实战总结 + PM 反馈修订。
> 原则：先修传输层，再扩并行；先改规则，不改组织图。

## 一、当前问题分析

### 1.1 今日工作量分布（PM=1.0 基准）

| 角色 | 工作量 | 问题 |
|------|--------|------|
| be-1（后端） | 2.5 | 严重过载，全天几乎不停 |
| pm-chief（PM） | 1.5 | 过度深入技术设计和代码细节 |
| review-1（审查） | 1.2 | 唯一审查者，串行瓶颈 |
| fe-1（前端） | 0.3 | 利用率低（但不宜急于移除） |
| qa-1（测试） | 0.3 | 等待时间长，实际工作少 |
| arch-1（架构师） | 0.0 | 全天闲置 |

### 1.2 核心问题

1. **角色职责错位**：PM 做了技术方案设计，arch-1 闲置
2. **审查单点瓶颈**：所有任务串行走 review-1，阻塞整条链路
3. **流程全串行**：分析→设计→开发→审查→测试→验收，无任何并行环节
4. **消息投递不可靠**：Codex 的 `i` 模式导致消息经常未送达（**传输层问题，非组织问题**）

## 二、职责边界调整

### 2.1 团队结构（当前阶段不变）

| Session | 角色 | 客户端 | 职责 |
|---------|------|--------|------|
| pm-chief | PM | Codex (omx) | **需求分诊**、任务拆解、派发、仲裁、验收。不做技术设计，不碰代码 |
| arch-1 | 架构师 | Codex (omx) | 复杂任务的需求分析、技术方案设计、接口契约定义、复杂任务设计审查 |
| be-1 | 后端开发 | Codex (omx) | 后端/API/数据库任务执行 |
| fe-1 | 前端开发 | Claude Code | 前端/组件/样式任务执行 |
| qa-1 | 测试 | Claude Code | 功能验证、回归测试、测试用例设计 |
| review-1 | 审查 | Codex (omx) | 代码质量审查 |

### 2.2 关键变更

- **PM 保留需求分诊权**：判断问题优先级、归类问题归属、决定派给谁。但**退出技术方案设计**，这部分交给 arch-1
- **arch-1 承接技术设计**：复杂任务由 arch-1 先出方案，PM 基于方案拆任务
- **fe-1 暂时保留**：不急于转为全栈，观察 1-2 天再评估
- **暂不新增 be-2**：先用现有团队跑 1-2 轮，数据证明后再扩组织

### 2.3 PM vs arch-1 的分工边界

| 环节 | PM | arch-1 |
|------|-----|--------|
| 看到生产问题 | ✅ 需求分诊：归类、定优先级 | |
| 复杂任务 | → 派给 arch-1 出方案 | ✅ 需求分析 + 技术方案 + 接口契约 + 验收标准 + 测试要点 |
| 简单任务 | ✅ 直接拆解派发 | |
| 基于方案拆任务 | ✅ 拆子任务、定 write_scope | |
| 审查裁决 | ✅ 汇总审查意见，最终裁决 | ✅ 复杂任务设计审查 |

## 三、消息投递可靠性（P0 前置）

> 这是所有并行化的前提。传输层不可靠，并行越多越乱。

### 3.1 当前问题

- Codex（omx）需要 `i` 进入插入模式才能输入，但 send-keys 时序不确定
- 多条消息快速发送时，`i` 被当成文本插入
- 没有投递确认机制

### 3.2 解决方案：统一 tmux 投递封装

```bash
# scripts/send-to-agent.sh <session> <message>
# 自动判断客户端类型（Codex/Claude Code），处理 i 模式，带重试
```

封装逻辑：
1. 判断 session 是 Codex 还是 Claude Code（从 config.json 读取）
2. Codex：先 `i`，sleep 0.5，发送消息，sleep 0.1，Enter
3. Claude Code：直接发送消息，Enter
4. 发送后 sleep 3 秒，检查 pane 是否出现 Working/⏺ 等响应标记
5. 无响应则重试一次（最多重试 1 次）

### 3.3 task-watcher 和 PM 统一使用此封装

所有 tmux send-keys 的地方改为调用 `send-to-agent.sh`，消除各处不一致的行为。

## 四、审查机制设计

### 4.1 审查分级

| review_level | 触发条件 | 审查者 | 审查方式 |
|--------------|---------|--------|----------|
| `skip` | 样式调整、文案修改、配置变更 | 无 | PM 直接验收 |
| `standard` | Bug 修复、小功能、重构 | review-1 | 单审 |
| `complex` | 新功能、架构变更、跨模块改动 | review-1 + arch-1 | 双审并行 |

PM 在创建任务时根据任务复杂度指定 `review_level`。双审仅限复杂任务。

### 4.2 双审分工

| 审查者 | 审查视角 | 关注点 |
|--------|---------|--------|
| **review-1** | 质量审查 | 代码规范、边界 case、测试覆盖、安全风险、可维护性 |
| **arch-1** | 设计审查 | 是否符合技术方案、接口契约一致性、架构影响、性能影响 |

两个维度互补，不重叠。

### 4.3 双审流程

```
be-1/fe-1 完成任务
    │
    ├──→ review-1 写 review.md（质量视角）
    │
    ├──→ arch-1 写 design-review.md（设计视角）
    │
    └──→ PM 读取两份审查，按规则裁决
```

### 4.4 冲突处理规则

| 情况 | 处理 |
|------|------|
| 双方都通过 | 推进 qa |
| 一方通过、一方有建议（非阻塞） | 记录建议，不阻塞，推进 |
| 一方通过、一方阻塞 | PM 判断，通常尊重阻塞方 |
| 双方都阻塞 | PM 裁决，必要时升级林总工 |
| 双方意见冲突 | PM 裁决，写明理由 |

### 4.5 时间控制（修订）

- 两个审查者**并行审查**，不串行等待
- 审查 deadline：标准任务 30 分钟，复杂任务 60 分钟
- **超时未提交 → 通知 PM，由 PM 决定如何处理**（不自动视为通过）
- task-watcher 监控审查超时

### 4.6 task.json 扩展字段

```json
{
  "review_level": "complex",
  "reviewers": ["review-1", "arch-1"],
  "review_gate": "all_pass",
  "review_deadline": "2026-04-24T20:00:00+08:00"
}
```

## 五、并行化流程设计

### 5.1 优化后流程（有约束的并行）

```
阶段1：需求分析
┌─────────────────────────────────────────────────┐
│ PM 需求分诊（快）                                │
│   ├── 简单任务 → 直接拆解派发                    │
│   └── 复杂任务 → arch-1 出方案                   │
└─────────────────────────────────────────────────┘

阶段2：开发执行
┌─────────────────────────────────────────────────┐
│ be-1 做后端任务    │  fe-1 做前端任务            │
│ （write_scope 不重叠）                           │
└─────────────────────────────────────────────────┘
                    ↕ 并行
┌─────────────────────────────────────────────────┐
│ qa-1 预写测试设计（基于 arch-1 的测试要点）       │
└─────────────────────────────────────────────────┘

阶段3：审查（分级）
┌─────────────────────────────────────────────────┐
│ skip → PM 直接验收                               │
│ standard → review-1 审查                         │
│ complex → review-1 + arch-1 并行双审             │
└─────────────────────────────────────────────────┘

阶段4：验证 → PM 裁决验收
```

### 5.2 并行化约束

1. **write_scope 不重叠**：批量派发前，PM 必须确保各任务的 write_scope 无交集
2. **共享文件归属明确**：PM 在 instruction 中标明"谁能改共享文件"
3. **测试前置是测试设计，不是测试代码**：qa-1 提前写测试用例文档，代码完成后再写测试代码
4. **review-1 审集成态 clean diff**：不审中间过程，只审最终结果

## 六、task-watcher 适配

### 6.1 新增监控点

| 事件 | 行为 |
|------|------|
| design-review.md 出现 | 检测 arch-1 的设计审查结论 |
| 双方审查都到齐 | 汇总通知 PM 裁决 |
| 审查超时 | **通知 PM，由 PM 决定**（不自动通过） |
| review_level=skip | 跳过审查，result 后直接推进 qa |

### 6.2 双审汇总通知 PM 格式

```
[task-watcher] 任务"xxx"双审完成：
- review-1（质量）：✅ 通过 / ❌ 阻塞（原因...）
- arch-1（设计）：✅ 通过 / ❌ 阻塞（原因...）
- ⚠️ arch-1 审查超时，请决定如何处理
请裁决是否推进。
```

## 七、arch-1 方案输出规范

arch-1 的方案文档（instruction.md 的基础）必须包含：

```markdown
# 技术方案：{任务标题}

## 需求分析
（从生产问题/需求中提取的关键信息）

## 技术方案
（实现思路、技术选型、关键代码路径）

## 接口契约
（API 变更、数据结构变更、前后端约定）

## 验收标准
（明确的、可验证的完成条件）

## 测试要点
（关键测试场景、边界 case、回归范围）

## write_scope
（允许修改的文件列表，确保与其他任务不重叠）

## 风险评估
（潜在风险、回滚方案）
```

## 八、实施计划

### P0：先改规则，不改组织图（立即可做）

1. **统一 tmux 投递封装**：`scripts/send-to-agent.sh`，所有消息发送统一走此脚本
2. **task.json schema 增加 review_level、reviewers、review_deadline**
3. **task-watcher 增加 design-review.md 检测、双审汇总、审查超时通知**
4. **明确 PM / arch-1 角色边界**：更新 AGENT.md
   - PM：需求分诊 + 拆解派发 + 仲裁验收，**退出技术设计**
   - arch-1：复杂任务的需求分析 + 技术方案 + 设计审查
5. **create-task.sh / dispatch-task.sh 增加 write_scope 冲突检查**：批量派发前自动检测与其他活跃任务的 write_scope 是否重叠，重叠则警告或阻止

### P1：试运行 1-2 轮（不改组织图）

1. 用现有团队（6 人）跑 1-2 天
2. 验证：
   - arch-1 是否真正吃到分析工作
   - review-1 审查分级是否降压
   - PM 是否从代码细节中退出
   - 消息投递是否可靠

### P1.5：引入 be-2（满足前置条件后）

满足以下全部条件后，引入 be-2 试运行：
1. 统一投递封装可用且稳定
2. write_scope 冲突检查已上线
3. review_level 分级已启用
4. arch-1 已开始承担复杂任务方案输出

### P2：数据验证后深化

1. 看板系统上线，量化各角色利用率和任务耗时
2. 评估 fe-1 是否转全栈（基于前端任务占比数据）
3. 考虑自动化流水线（CI/CD 集成）
