## 三、消息投递可靠性（P0 前置）

> 这是所有并行化的前提。传输层不可靠，并行越多越乱。

### 3.1 当前问题

- Codex（omx）需要 `i` 进入插入模式才能输入，但 send-keys 时序不确定
- 多条消息快速发送时，`i` 被当成文本插入
- 没有投递确认机制

### 3.2 解决方案：统一 tmux 投递封装

```bash
# scripts/send-to-agent.sh <session> <message>
# 自动判断客户端类型（Codex/Claude Code），处理 i 模式，带重试
```

封装逻辑：
1. 判断 session 是 Codex 还是 Claude Code（从 config.json 读取）
2. Codex：先 `i`，sleep 0.5，发送消息，sleep 0.1，Enter
3. Claude Code：直接发送消息，Enter
4. 发送后 sleep 3 秒，检查 pane 是否出现 Working/⏺ 等响应标记
5. 无响应则重试一次（最多重试 1 次）

### 3.3 task-watcher 和 PM 统一使用此封装

所有 tmux send-keys 的地方改为调用 `send-to-agent.sh`，消除各处不一致的行为。

## 四、审查机制设计

### 4.1 审查分级

| review_level | 触发条件 | 审查者 | 审查方式 |
|--------------|---------|--------|----------|
| `skip` | 样式调整、文案修改、配置变更 | 无 | PM 直接验收 |
| `standard` | Bug 修复、小功能、重构 | review-1 | 单审 |
| `complex` | 新功能、架构变更、跨模块改动 | review-1 + arch-1 | 双审并行 |

PM 在创建任务时根据任务复杂度指定 `review_level`。双审仅限复杂任务。

### 4.2 双审分工

| 审查者 | 审查视角 | 关注点 |
|--------|---------|--------|
| **review-1** | 质量审查 | 代码规范、边界 case、测试覆盖、安全风险、可维护性 |
| **arch-1** | 设计审查 | 是否符合技术方案、接口契约一致性、架构影响、性能影响 |

两个维度互补，不重叠。

### 4.3 双审流程

```
dev-2/dev-1 完成任务
    │
    ├──→ review-1 写 review.md（质量视角）
    │
    ├──→ arch-1 写 design-review.md（设计视角）
    │
    └──→ PM 读取两份审查，按规则裁决
```

### 4.4 冲突处理规则

| 情况 | 处理 |
|------|------|
| 双方都通过 | 推进 qa |
| 一方通过、一方有建议（非阻塞） | 记录建议，不阻塞，推进 |
| 一方通过、一方阻塞 | PM 判断，通常尊重阻塞方 |
| 双方都阻塞 | PM 裁决，必要时升级林总工 |
| 双方意见冲突 | PM 裁决，写明理由 |

### 4.5 时间控制（修订）

- 两个审查者**并行审查**，不串行等待
- 审查 deadline：标准任务 30 分钟，复杂任务 60 分钟
- **超时未提交 → 通知 PM，由 PM 决定如何处理**（不自动视为通过）
- task-watcher 监控审查超时

### 4.6 task.json 扩展字段

```json
{
  "review_level": "complex",
  "reviewers": ["review-1", "arch-1"],
  "review_gate": "all_pass",
  "review_deadline": "2026-04-24T20:00:00+08:00"
}
```

## 五、并行化流程设计

### 5.1 优化后流程（有约束的并行）

```
阶段1：需求分析
┌─────────────────────────────────────────────────┐
│ PM 需求分诊（快）                                │
│   ├── 简单任务 → 直接拆解派发                    │
│   └── 复杂任务 → arch-1 出方案                   │
└─────────────────────────────────────────────────┘

阶段2：开发执行
┌─────────────────────────────────────────────────┐
│ dev-2 做后端任务    │  dev-1 做前端任务            │
│ （write_scope 不重叠）                           │
└─────────────────────────────────────────────────┘
                    ↕ 并行
┌─────────────────────────────────────────────────┐
│ qa-1 预写测试设计（基于 arch-1 的测试要点）       │
└─────────────────────────────────────────────────┘

阶段3：审查（分级）
┌─────────────────────────────────────────────────┐
│ skip → PM 直接验收                               │
│ standard → review-1 审查                         │
│ complex → review-1 + arch-1 并行双审             │
└─────────────────────────────────────────────────┘

阶段4：验证 → PM 裁决验收
```

### 5.2 并行化约束

1. **write_scope 不重叠**：批量派发前，PM 必须确保各任务的 write_scope 无交集
2. **共享文件归属明确**：PM 在 instruction 中标明"谁能改共享文件"
3. **测试前置是测试设计，不是测试代码**：qa-1 提前写测试用例文档，代码完成后再写测试代码
4. **review-1 审集成态 clean diff**：不审中间过程，只审最终结果

## 八、任务粒度与调度效率规则（v2.1 新增）

> 第二日实战暴露的核心问题：PM 对所有问题都按"复杂流程"处理，导致协调开销超过执行开销。

### 8.1 任务粒度判断规则

PM 在拆解任务前，必须先判断任务粒度：

| 粒度 | 判断标准 | 处理方式 | 示例 |
|------|---------|---------|------|
| **微型** | 一个人 5 分钟内能定位和解决 | 直接派给一个 agent，不拆子任务 | 看板统计不对、配置项缺失、文案修正 |
| **小型** | 一个人 30 分钟内能完成 | 派给一个 agent，不拆子任务 | 简单 bug 修复、单个接口调整 |
| **中型** | 需要 1-2 个 agent 协作 | 可拆 2-3 个子任务，但由 PM 直接管理 | 前后端联调、单功能开发 |
| **大型** | 跨模块、需要方案设计 | 按 epic/domain 流程走，arch-1 出方案 | 新功能开发、架构变更 |

**关键原则：一个人能搞定的事，不要拆成两个人的任务。**

### 8.2 排查类任务的特殊规则

排查类任务（看板数据不对、功能异常、性能问题）适用以下规则：

1. **先派一个人定位问题，不要同时派前后端**
2. 定位后确认根因在哪一侧，再派对应的 agent 修复
3. 如果 15 分钟内无法定位，PM 可以增派第二个 agent 协助
4. 排查任务的 write_scope 应覆盖"问题可能涉及的文件范围"，而不是精确到某个文件

**错误做法：** "看板统计不对" → 同时派 dev-1 查前端 + dev-2 查后端
**正确做法：** "看板统计不对" → 先派 dev-2 排查（因为统计是后端 API 返回的）→ 定位后再决定

### 8.3 PM 上下文管理

PM 上下文是整个团队的瓶颈资源，需要珍惜：

| 行为 | 消耗 | 建议 |
|------|------|------|
| 每个子任务都写 instruction.md | 高 | 微型/小型任务可以直接用 send-to-agent.sh 发一句话指令 |
| 每个任务都等确认再推进 | 高 | 补修、QA、收口等环节自主推进 |
| 同时跟踪 >5 个活跃任务 | 高 | 控制并发任务数，完成一批再开下一批 |
| 状态汇报写长文 | 中 | 用结构化简报（状态+阻塞+下一步） |

**PM 应该定期 compact，但 compact 前必须确保当前活跃任务的关键信息已落盘到 task.json。**

### 8.4 角色越界约束

每个 agent 只能执行自己角色范围内的任务：

| 角色 | 只能做的事 | 禁止做的事 |
|------|-----------|-----------|
| **pm-chief** | 需求分诊、任务拆解派发、状态跟踪、仲裁验收 | 直接写业务代码、部署生产、执行测试 |
| **arch-1** | 技术方案设计、接口契约定义、设计审查 | 需求分诊、写业务代码、执行测试 |
| **dev-2** | 后端/API/数据库代码实现 | 前端代码、任务拆解、审查裁决 |
| **dev-1** | 前端/组件/样式代码实现 | 后端代码、任务拆解、审查裁决 |
| **review-1** | 代码质量审查、设计审查 | 执行测试、写业务代码、部署 |
| **qa-1** | 功能验证、回归测试、测试用例设计 | 写业务代码、审查裁决 |

**违反处理：** 如果 PM 发现某个 agent 在执行超出角色范围的任务（如 review-1 做了 QA 的事），应在审查裁决中标注，并通知该 agent 的指导文件。

## 九、方案确认门（v2.1 新增）

> 解决"实现出来和预期偏离"的问题。

### 9.1 确认门流程

```
arch-1 完成技术方案
    │
    ├──→ PM 整理方案摘要（目标、方案要点、子任务列表、风险评估）
```

| 场景 | 需要确认？ | 原因 |
|------|-----------|------|

## 十、事件驱动自动流转（v2.2 待实施）

> 目标：PM 介入从 7 次降到 2 次（需求分诊 + 方案确认），其余流程节点自动流转。

### 10.1 当前串行链路 vs 目标链路

**当前（PM 介入 7 次）：**
```
PM 需求分析 → arch-1 出方案 → PM 飞书确认 → PM 拆任务 → PM 派 dev → dev 开发
→ PM 派 review → review-1 审查 → PM 裁决 → PM 派 QA → qa-1 测试 → PM 收口
```

**目标（PM 介入 2 次）：**
```
PM 需求分诊 → arch-1 出方案 → [自动通知 PM 推送确认] → 林总工确认
→ PM 批量创建子任务 → dev 并行开发
→ [自动] review-1 审查 → [自动] qa-1 测试 → [自动] 收口
（PM 只在 review 驳回 / QA 失败时介入仲裁）
```

### 10.2 task-watcher 新增自动流转规则

| 检测事件 | 自动动作 | PM 是否介入 |
|---------|---------|------------|
| arch-1 的 domain/epic 任务出现 result.json | 通知 PM 推送方案给林总工确认 | ✅ PM 转达 |
| execution 任务出现 result.json 且 review_level=standard | 自动通知 review-1 审查 | ❌ |
| execution 任务出现 result.json 且 review_level=complex | 同时通知 review-1 + arch-1 双审 | ❌ |
| review-1 通过（review.md 标记 approved） | 自动通知 qa-1 测试 | ❌ |
| qa-1 通过（verify.json 标记 pass） | 自动执行 close-task.sh 收口 | ❌ |
| review 驳回 或 QA 失败 | 通知 PM 仲裁并决定补修方案 | ✅ |

### 10.3 共享项目知识库

**问题：** PM 每次在 instruction.md 里重复写项目约定（编码规范、API 风格、数据库命名等），浪费 token 和时间。

**方案：** 在 `prompts/` 目录下维护共享知识文件：

```
prompts/
  coding-standards.md   # 编码规范、命名约定、注释风格
  api-conventions.md    # API 设计模式、错误码规范、分页约定
  db-conventions.md     # 数据库命名、迁移规范
  frontend-patterns.md  # 前端组件模式、状态管理约定
```

- 所有 dev agent 启动时自动读取这些文件
- PM 在 instruction.md 里只需写任务特有的内容，不需要重复项目约定
- arch-1 方案中引用共享知识文件中的规范，而不是重新描述

### 10.4 并行开发优化

**批量创建 + 自动认领：**
- PM 确认方案后，一次性创建所有子任务（不再逐个派发）
- 子任务状态设为 `pending`，dev agent 可以自行认领
- task-watcher 检测到 `pending` 任务时，通知空闲 dev 认领

**arch-1 方案并行：**
- 如果有多个独立需求，PM 可以同时创建多个 domain 任务给 arch-1
- arch-1 可以并行处理不相关的需求方案
- 每个方案独立通知 PM 推送确认

### 10.5 实施优先级

| 项目 | 优先级 | 依赖 |
|------|--------|------|
| 方案完成自动通知 PM | P0 | task-watcher 已有基础 |
| review 通过自动通知 QA | P0 | task-watcher 已有基础 |
| QA 通过自动收口 | P0 | close-task.sh 已有 |
| 共享项目知识库 | P1 | 无 |
| 批量创建 + 自动认领 | P1 | 需要认领机制 |
| review 驳回/QA 失败通知 PM | P0 | task-watcher 已有基础 |
