# 任务协作看板优化方案

> 创建时间：2026-05-03  
> 适用仓库：`~/Desktop/work/my-agent-teams/`  
> 目标：在现有任务看板基础上，增强**任务进展可视化**、**沟通过程串联**、**后续数据分析能力**。

---

## 一、背景与目标

林总工对现有任务看板提出三类新需求：

1. **方便查看任务进展**
   - 能快速看每个任务当前处于哪个状态
   - 能看到状态流转路径、各阶段耗时、当前责任角色
   - 能快速判断任务卡在谁、卡在哪一步

2. **方便查看角色沟通过程**
   - 把 task-watcher 通知、PM 派发、Chat Hub 讨论、review / QA 过程串起来
   - 以“任务”为主线，而不是以零散日志为主线
   - 能做时间线回放和责任链追溯

3. **方便后续做数据分析**
   - 任务完成率、平均耗时、返工率、阻塞率
   - 各角色负载与效率
   - 常见瓶颈（例如 review 堵塞、QA 排队、PM 派发延迟）

本方案以**在现有 dashboard 基础上迭代**为前提，不另起一套新系统。

## 2.2 当前实现的不足

虽然基础能力已具备，但对新需求还不够：

### A. 任务进展可视化仍偏“摘要态”
当前看板能看到：
- 当前列
- 部分时间点

但还缺：
- 单任务完整状态流转明细
- 每一段耗时拆解（例如 dispatched→ack、ack→result、result→review、review→verify）
- 当前负责角色视图
- “为什么 blocked / 为什么返工”的结构化原因聚合

### B. 沟通过程没有进入看板主模型
当前 SQLite 只 ingest：
- `tasks/` 工件
- `transitions.jsonl`

但没有 ingest：
- `chat/general/*.jsonl`
- `chat/tasks/*.jsonl`
- PM 派发消息本身的结构化记录
- watcher 通知的统一事件流

结果是：
- 任务状态看得到
- 任务讨论过程看不到
- 无法做“任务 timeline + 通信 timeline”合并展示

### C. 分析能力还不够成体系
当前 agent 统计偏展示，缺少分析口径：
- 完成率
- 平均耗时
- 阻塞率
- 返工率
- review / QA 延迟
- 角色负载饱和度
- domain / task_level / owner_pm 维度聚合

### D. 数据模型还没覆盖“事件中心化”
当前的 `task_events` 更偏任务工件事件，不是完整协作事件流。
要满足新需求，需要把：
- 状态事件
- 通信事件
- 通知事件
- 分析衍生指标
统一建模。

## 三、总体设计方向

## 3.1 核心原则

1. **在现有 dashboard 上增量演进，不推倒重做**
2. **任务仍是事实源中心**，但看板要升级成“任务 + 沟通 + 指标”的组合视图
3. **先做事件统一，再做复杂图表**
4. **先做任务详情页，再做跨任务分析页**
5. **SQLite 仍可继续用作 Phase 1/2 数据底座**

---

## 3.2 推荐形态

推荐继续采用 **Web Dashboard**，而不是 CLI dashboard。

原因：
- 现有 Flask + ECharts 已经在跑
- 甘特图、时间线、任务详情抽屉、角色对比图更适合 Web
- 后续如果要增加 drill-down、筛选器、图表联动，Web 成本更低

CLI 可保留为辅助巡检工具，但不作为主看板。

---

## 四、功能模块设计

建议把看板升级为 **六个功能模块**。

---

## 4.1 模块一：任务总览（Overview）

### 目标
让 PM / 林总工一眼看到当前全局进展。

### 展示内容
- 总任务数
- 活跃任务数
- blocked 数
- ready_for_merge 数
- done 数
- 近 24h 新建 / 完成 / 阻塞数量
- 当前最久未推进任务 TOP N
- 当前最重负载角色 / agent

### 页面位置
- 首页顶部 summary cards
- 当前 Kanban 页顶部加入统计条

---

## 4.2 模块二：任务进展看板（Task Board）

### 目标
保留现有 Kanban，但增强任务卡片信息密度。

### 每张任务卡建议增加
- 任务标题
- 当前状态 / board_status
- 当前责任人（assigned_agent / reviewer / QA / integration_owner）
- 当前角色阶段（开发 / 审查 / QA / 集成 / 部署）
- 创建时间
- 当前状态停留时长
- 是否 critical / prod / blocked
- 是否超 SLA

### 交互
- 点击任务卡，打开 **任务详情抽屉 / Modal**
- 支持按以下维度过滤：
  - project
  - domain
  - owner_pm
  - assigned_agent
  - review_level
  - target_environment
  - priority

---

## 4.3 模块三：任务详情页 / 抽屉（Task Detail）

### 目标
这是本轮最关键的新模块。

### 展示内容分 4 个区块

#### A. 基本信息
- task_id / title
- project / domain / task_level
- owner_pm / assigned_agent / reviewer / integration_owner
- priority / target_environment / review_level
- depends_on / blocks
- write_scope

#### B. 状态流转时间线
按时间排序展示：
- created
- dispatched
- ack
- result
- review
- verify
- done / blocked / cancelled / timeout

每一条显示：
- 时间
- 事件类型
- from → to
- reason
- source

#### C. 阶段耗时拆解
自动计算：
- 创建 → 派发
- 派发 → ACK
- ACK → 结果
- 结果 → 审查
- 审查 → 验证
- 验证 → 收口

#### D. 沟通过程时间线
按任务维度串联：
- `task_announce`
- `question / answer`
- `decision`
- `task_done`
- watcher 自动通知
- PM 定向唤醒

这部分是“协作链路可视化”的核心。

---

## 4.4 模块四：沟通与通知视图（Communication View）

### 目标
让用户按任务或按角色查看沟通过程，而不是只看状态结果。

### 两种视角

#### 视角 A：按任务
展示某任务的：
- Chat thread
- PM 派发消息
- task-watcher 自动通知
- review / QA / integration 触发消息

#### 视角 B：按角色
展示某 agent / PM / reviewer 的：
- 最近参与过哪些任务
- 发过哪些 question / answer / decision
- 收到过哪些 watcher 通知
- 是否存在长时间未响应 thread

### 页面布局建议
- 左侧：任务列表 / 角色列表
- 右侧：时间线流
- 顶部：筛选器（日期范围、任务、角色、事件类型、severity）

---

## 4.5 模块五：角色效率与负载分析（Role Analytics）

### 目标
回答“谁最忙、谁最堵、哪个环节最慢”。

### 指标建议

#### Agent 级
- active_task_count
- completed_task_count
- blocked_task_count
- average_cycle_time
- average_ack_delay
- average_review_wait
- average_verify_wait

#### Role 级
- PM 派发量
- reviewer 审查吞吐
- QA 验证吞吐
- integration 任务吞吐

#### 组织级
- domain 完成率
- task_level 分布
- owner_pm 负载
- blocked 原因分布

### 图表建议
- 柱状图：完成数 / 当前负载
- 箱线图：任务耗时分布
- 热力图：角色 × 状态停留时长
- 漏斗图：dispatched → ack → result → review → verify → done

---

## 4.6 模块六：瓶颈与异常识别（Bottleneck / Alerts）

### 目标
让看板能主动指出风险，而不是只被动展示数据。

### 识别规则建议
- dispatched 超过阈值未 ACK
- working 超过阈值未 result
- ready_for_merge 卡住过久
- review 循环次数过多
- verify fail 反复出现
- 某 domain blocked 占比过高
- 某角色当前负载超过阈值
- critical 任务没有双通道通知记录

### 页面形式
- 首页右侧异常面板
- 或独立的 “Alerts / 风险” Tab

## 六、数据模型设计

目标：从当前 `tasks/` 和 `chat/` 抽出一套支持“任务 + 沟通 + 分析”的结构化模型。

---

## 6.1 现有可复用表

### 已有：`tasks`
当前已支持：
- task_id
- title
- project
- domain
- assigned_agent
- reviewer
- owner_pm
- parent_task_id
- root_request_id
- current_status
- board_status
- created_at / dispatched_at / ack_at / completed_at / review_completed_at / verify_completed_at / current_status_at
- summary
- review_state
- verify_ok
- write_scope_json
- artifacts_json

### 已有：`task_events`
当前已支持：
- created
- status_transition
- ack
- result
- review_completed
- verify_completed

这两个表足够作为第一层任务事实基础。

---

## 6.2 建议新增表：`communication_events`

### 用途
统一 ingest：
- `chat/general/*.jsonl`
- `chat/tasks/*.jsonl`
- PM 派发消息
- watcher 通知消息
- send-to-agent 强制唤醒消息

### 建议字段
```sql
communication_events (
  event_id TEXT PRIMARY KEY,
  task_id TEXT,
  thread_id TEXT,
  channel TEXT,              -- general / task / watcher / direct_nudge
  event_type TEXT,           -- task_announce / question / answer / decision / task_done / notify
  source_type TEXT,          -- human / system
  from_actor TEXT,
  to_actor TEXT,
  severity TEXT,             -- info / degraded / critical
  priority TEXT,             -- low / medium / high / critical
  message_text TEXT,
  reply_to TEXT,
  related_artifact_path TEXT,
  happened_at TEXT,
  observed_at TEXT,
  payload_json TEXT
)
```

### 说明
- `task_id` 为空时表示 general 频道公共事件
- `thread_id` 支持 task 线程聚合
- `severity` 为后续误报警降噪做准备

---

## 6.3 建议新增表：`task_metrics_daily`

### 用途
存每天的聚合指标，减少每次页面打开都全量扫描 JSONL。

### 建议字段
```sql
task_metrics_daily (
  stat_date TEXT,
  project TEXT,
  metric_key TEXT,
  metric_value REAL,
  payload_json TEXT,
  PRIMARY KEY (stat_date, project, metric_key)
)
```

### 指标示例
- task_created_count
- task_done_count
- blocked_count
- completion_rate
- avg_cycle_time_hours
- avg_ack_delay_minutes
- avg_review_wait_minutes
- critical_dual_channel_count
- unanswered_question_count

---

## 6.4 建议新增表：`agent_metrics_daily`

### 用途
支撑角色效率与负载分析。

### 建议字段
```sql
agent_metrics_daily (
  stat_date TEXT,
  agent_id TEXT,
  role TEXT,
  active_task_count INTEGER,
  completed_task_count INTEGER,
  blocked_task_count INTEGER,
  avg_cycle_time_hours REAL,
  avg_response_delay_minutes REAL,
  total_tracked_work_seconds REAL,
  payload_json TEXT,
  PRIMARY KEY (stat_date, agent_id)
)
```

---

## 6.5 建议新增表：`task_stage_durations`

### 用途
把各阶段耗时结构化，便于后续直接画图。

### 建议字段
```sql
task_stage_durations (
  task_id TEXT PRIMARY KEY,
  create_to_dispatch_seconds REAL,
  dispatch_to_ack_seconds REAL,
  ack_to_result_seconds REAL,
  result_to_review_seconds REAL,
  review_to_verify_seconds REAL,
  verify_to_close_seconds REAL,
  total_cycle_seconds REAL,
  updated_at TEXT
)
```

---

## 七、推荐实施方案

建议分三期。

---

## 7.1 第一期：补齐事件模型（P0/P1）

### 目标
先让看板能串起“任务 + 通信”。

### 工作项
1. 扩展 ingest：加入 `communication_events`
2. 为 `chat/` 建立 ingest 脚本
3. 给 `task-watcher` / `dispatch-task.sh` / `send-to-agent.sh` 留下统一事件痕迹
4. 新增任务详情页，展示：
   - 状态时间线
   - 沟通时间线
   - 阶段耗时

### 验收
- 能打开某个任务，完整看到：
  - 派发
  - ack
  - result
  - review
  - verify
  - chat 讨论
  - watcher 通知

---

## 7.2 第二期：补齐分析面板（P2）

### 目标
让看板开始回答效率与瓶颈问题。

### 工作项
1. 日指标脚本
2. 角色指标脚本
3. 只读聚合视图（owner_pm/domain/task_level/parent_task_id）
4. 异常识别面板

### 验收
- 能看完成率、平均耗时、角色负载、domain 堵塞情况

---

## 7.3 第三期：风险与治理增强（P3）

### 目标
让系统从“可看”升级到“可预警”。

### 工作项
1. severity 分级接入 dashboard
2. critical / degraded 事件分层展示
3. 风险榜与瓶颈榜
4. 看板过滤器增强（按 severity / owner_pm / role）

### 验收
- PM / 林总工能一眼看到：
  - 真正需要打断处理的 critical
  - 普通退化类噪音已被降级聚合

## 十、最终建议

### 推荐路线
- **不重做**，直接在现有 `dashboard/` 上扩展
- **先统一事件模型**，再做复杂图表
- **先做任务详情 + 沟通串联**，再做跨任务分析
