# 任务协作看板 - 任务拆解

> 创建时间：2026-05-03  
> 对应方案：`design/任务协作看板优化方案.md`  
> 目标：把看板优化方案拆成可直接派发给 `dev-1 / dev-2 / arch-1 / qa-1` 的执行任务列表。  
> 原则：**第一期拆得最细，第二期与第三期先保持粗粒度。**

---

## 一、拆解说明

### 1.1 角色分工假设
- `arch-1`
  - 负责方案细化、数据模型边界、跨模块集成设计、接口契约把关
- `dev-1`
  - 偏前端 / Flask 页面 / dashboard UI / JS 展示层
- `dev-2`
  - 偏后端 / SQLite ingest / query / sync 脚本 / 数据层
- `qa-1`
  - 负责接口验证、数据一致性验证、页面联调验证、回归测试方案

### 1.2 分期原则
- **第一期**：任务详情 + 沟通事件 + 时间线，拆到可以直接开工
- **第二期**：分析面板与只读聚合，先按能力域拆
- **第三期**：严重度、风险识别、预警增强，先按粗粒度拆

## PH1-05 把 watcher / 派发 / 强制唤醒消息纳入 communication_events
- **任务标题**：统一记录 task-watcher / PM 派发 / send-to-agent 强制唤醒事件
- **指派角色**：`dev-2`
- **前置依赖**：`PH1-01`、`PH1-02`

  1. 设计一种统一落盘入口（不要让 3 个脚本各自发明事件格式），让下列事件进入看板数据层：
     - `dispatch-task.sh` 成功派发
     - `send-to-agent.sh` 成功/失败/重试
     - `task-watcher.sh` 的 review/QA/PM 通知
  2. 可选实现：
     - 写入统一 JSONL 再 ingest

- **涉及文件**：
  - `scripts/dispatch-task.sh`

  - 如有必要：新增 `scripts/chat-board-sync.py`
- **验收标准**：

# 六、最终建议

### 最值得先做的 4 个任务
如果只先开最小闭环，建议优先派：
1. `PH1-02` 扩 SQLite schema
2. `PH1-03` 实现 chat ingest
3. `PH1-08` 实现任务详情/时间线 API
4. `PH1-10 ~ PH1-12` 实现任务详情页 + 沟通时间线

### 一句话结论
> 第一阶段最关键的不是“再多一张统计图”，而是先把单任务的**状态链路、沟通链路、耗时链路**串成同一个详情视图；只要这一步做对，后面的分析与预警才有可靠底座。
