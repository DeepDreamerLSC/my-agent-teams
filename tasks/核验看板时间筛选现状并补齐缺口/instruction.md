# 任务：核验看板时间筛选现状并补齐缺口

## 任务类型
开发 / 前后端遗留吸收

## 目标
接管原 `增加看板时间筛选前端界面` 与 `增加看板时间筛选后端支持` 两个已取消但无人正式吸收的遗留任务，核验当前 dashboard 是否已经覆盖“任务看板时间范围筛选”原始目标；若有缺口则直接补齐，若无缺口则补出明确的归档说明与测试证据。

## 任务边界
- 只允许修改以下范围：
  - `dashboard/app.py`
  - `dashboard/query.py`
  - `dashboard/static/js/dashboard.js`
  - `dashboard/templates/index.html`
  - `dashboard/tests/test_dashboard_frontend_timeline.py`
  - `dashboard/tests/test_dashboard_metrics.py`
  - `design/task-board/system-design.md`
- 不修改任务池 / watcher / claim 相关脚本。
- 不重做整套看板 UI，只吸收“时间范围筛选”遗留缺口。

## 输入事实
- 原 fe-1 / be-1 遗留任务已取消并准备归档，但当前 PM 需要确认其需求是否已被现有实现覆盖。
- 现有代码中已经出现 gantt 快捷时间范围、`start_date` / `end_date` 参数与相关测试痕迹，需要你做完整串联核验。
- 若代码已覆盖原始需求，仍需把“原任务 → 当前实现/测试位置”的映射沉淀到设计文档，便于后续归档追溯。

## 约束
- 结论必须基于真实代码路径与自动化测试，不允许口头判定“应该已经有了”。
- 若发现前后端契约不一致，优先做最小改动补齐，不要额外扩 scope 做 unrelated 重构。
- 保持前端过滤交互、后端查询参数与测试断言口径一致。

## 交付物
1. 时间范围筛选现状核验结论。
2. 必要的前后端补丁与自动化测试。
3. `design/task-board/system-design.md` 中的遗留任务吸收/归档说明。

## 验收标准
- 能明确回答：原“看板时间筛选前端/后端”需求当前是否已落地。
- 如有缺口，补丁后测试通过，且前端/后端/测试三处口径一致。
- 如无缺口，文档中要能追溯到具体实现文件与测试文件。
- 输出中需显式说明原 fe-1 / be-1 遗留任务为什么可以归档。

## 下游动作
完成后，PM 将据此正式清空看板时间筛选遗留项，并把无人处理的 fe-1 / be-1 任务归档为“已吸收/已替代”。
